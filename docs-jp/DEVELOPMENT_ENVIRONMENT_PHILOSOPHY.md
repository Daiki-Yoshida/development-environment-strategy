## 開発環境は契約である

開発環境は、各開発者が手元へ好きにツールを入れて使うだけのものではありません。

開発作業をどこから始め、どこで実行し、どう分離し、どう検証し、最後にどう片付けるかを定めた「契約」として扱います。

```yaml
開発環境の契約:
  構造: "リポジトリ、基準チェックアウト、worktree、ツール、キャッシュ、生成物をどこへ置くか"
  ツール構成: "ホストへ置くツールと、プロジェクト実行環境へ置くツールの境界"
  公開コマンド: "人、AIエージェント、CIが利用する安定した操作"
  状態管理: "環境を作成、確認、再利用、削除、復旧する方法"
  分離: "プロジェクト、タスク、ブランチ、並列エージェント同士の衝突を防ぐ方法"
  安全性: "通常操作、破壊的操作、事前確認が必要な操作の区別"
```

内部のDocker構成やスクリプトは変更されても構いません。ただし、日常的に使うビルド、テスト、診断、削除の入口は、意味が明確で安定している必要があります。

## 優先順位

```yaml
優先順位:
  1: "ホストとデータの安全性"
  2: "再現性"
  3: "リポジトリとリソースの分離"
  4: "複数AIエージェントの並列運用"
  5: "操作内容と副作用の明確さ"
  6: "診断と復旧のしやすさ"
  7: "ローカルとCIの実行経路の一致"
  8: "開発効率"
```

開発効率も重要です。

安全性を理由に、日常作業を不必要に面倒にしてはいけません。安全な操作を短く分かりやすく提供し、それが最も使いやすい経路になるようにします。

## 制御面と実行面

```yaml
ホストの制御面:
  役割: "ソース管理、コンテナ操作、コマンド振り分け、認証、遠隔接続を管理する"
  原則: "明示的な例外がない限り、プロジェクト固有の実行環境やパッケージ環境を置かない"
コンテナの実行面:
  役割: "言語ランタイム、パッケージマネージャー、ビルド、テスト、プロジェクト固有CLIを持つ"
  原則: "リポジトリで管理された定義から、プロジェクト実行環境を再現できるようにする"
```

ホストは、原則としてプロジェクトの実行環境そのものではありません。プロジェクトの実行環境を制御する場所です。

この分離により、次の問題を減らします。

- ホスト環境が無秩序に変更される
- プロジェクト間でツールのバージョンが衝突する
- AIエージェントが必要以上の権限を使う
- 開発者やエージェントごとに実行結果が変わる

## ワークスペース構造の基本用語

```yaml
Workspace_Repository:
  意味: "開発ツール、ワークスペース全体の調整、環境文書、worktree管理を所有するリポジトリ"
Component_Repository:
  意味: "プロダクトコードと、そのプロダクトのGit履歴を所有する独立リポジトリ"
Primary_Checkout:
  意味: "同期、worktree作成、統合、最終確認に使用するComponent Repositoryの安定した基準チェックアウト"
Task_Worktree:
  意味: "一つのタスク、一つのブランチ、一つの書き込みエージェントへ割り当てる一時的なチェックアウト"
```

Workspace RepositoryとComponent Repositoryは、完全に別のGit履歴を持っていて構いません。

これは「開発ワークスペース上の関係」であり、Git submoduleであることを意味しません。

また、すべてのプロジェクトを必ず複数リポジトリに分ける必要もありません。開発環境とプロダクトコードが同じライフサイクルを持ち、分離の必要がなければ、一つのリポジトリで同じ原則を適用できます。

## 複数AIエージェントの分離

並列開発では、ブランチだけを分けても不十分です。

```yaml
タスク単位で分離するもの:
  - "ブランチ"
  - "作業ディレクトリ"
  - "コンテナの名前空間"
  - "ネットワークとホストポート"
  - "共有すると結果へ影響する可変ボリューム"
  - "ログと生成物"
原則: "一つのTask Worktreeを同時に書き込むAIエージェントは一つだけ"
```

安全に共有できる読み取り専用キャッシュは共有して構いません。しかし、テスト結果や実行結果を変える可能性がある状態を、効率だけを理由に共有してはいけません。

タスク、ブランチ、worktree、コンテナ、ログは、共通した安定識別子から追跡できるようにします。

## 操作内容を明確にする

コマンド名も開発環境契約の一部です。

```yaml
原則: "実行前に、操作対象と起こることを判断できなければならない"
必要なこと:
  - "意味が曖昧なライフサイクル操作には対象範囲を付ける"
  - "通常の後片付けと、データを捨てる強制削除を分ける"
  - "生のツール構文ではなく、安定した目的をコマンドとして公開する"
  - "失敗を隠さず、次に何を確認すべきか分かる出力を残す"
```

プロジェクト内で意味が一つに定まるなら、短いコマンド名でも構いません。ただし、短さを理由に意味を曖昧にしてはいけません。

## 再現性

あるリポジトリ状態を取得したとき、その開発環境の動作を再現できるだけの情報が、リポジトリ側に存在する必要があります。

```yaml
必要な性質:
  - "重要なツールのバージョンまたは許容範囲が管理されている"
  - "依存関係のlockファイルを尊重する"
  - "文書化されていないホスト状態へ依存しない"
  - "可能な限り、ローカルとCIが同じプロジェクト管理コマンドを使用する"
  - "外部Workspace Repositoryへ依存する場合、使用するバージョンの選び方を明示する"
```

再現性とは、永遠に更新しないことではありません。変更が意図的で、レビュー可能で、どの変更が結果へ影響したか追跡できることです。

## 安全性と使いやすさの両立

```yaml
原則: "安全な操作を最も簡単な操作にする"
方針:
  - "日常操作は、初期状態では非破壊的にする"
  - "破壊的操作は、名前と対象範囲を明確にする"
  - "診断コマンドを見つけやすくする"
  - "最終検証の標準経路を一つ定義し、文書化する"
  - "後片付けは、選択されたプロジェクトまたはタスクのリソースだけを対象にする"
```

何度も手作業や確認を要求すると、エージェントが安全経路を避ける原因になります。安全性は、コマンドの設計とリソース識別へ組み込みます。

## この戦略が担当する範囲

```yaml
担当する:
  - "ホストとコンテナの責務"
  - "開発ツールの実行方法"
  - "リポジトリとworktreeの構造"
  - "開発環境に関するトップレベルフォルダ"
  - "公開コマンドと開発環境用スクリプト"
  - "ローカルとCIの実行経路"
  - "環境状態、診断、後片付け、復旧"
担当しない:
  - "アプリケーションコードの設計"
  - "ドメインモジュールの境界"
  - "documents/ の案内やバージョン管理"
  - "Issue整理やPull Requestの承認方針"
  - "リリース統制やチーム権限"
```

スクリプト内部のコード品質は `design-principles` で評価します。スクリプトの配置と呼び出し方は、この戦略で評価します。`documents/` の構成は `documentation-strategy` で評価します。

## よくある誤解

- **Dockerファースト**は、すべての操作をDocker内で行うという意味ではない。GitやDocker操作など、制御面の処理はホストに置ける。
- **ホスト依存を最小化する**とは、全プロジェクト共通の固定許可リストを作ることではない。ツールの役割で判断する。
- **一つのTask Worktreeにつき一つのエージェント**とは、閲覧プロセスまで一つに制限する意味ではない。書き込み責任を一つにする。
- **Workspace Repository**は、Git上の親リポジトリや必須submoduleを意味しない。
- **Primary Checkout**は、通常の機能開発場所ではない。統合と管理のために安定させる。
- **リソース分離**は、すべてのキャッシュを複製する意味ではない。安全に共有できるキャッシュは共有できる。
- **明示的な名前**は、すべてのコマンドへ機械的に接頭辞を付ける意味ではない。意味や副作用が曖昧な場合に対象を示す。
- **再現性**は、更新を禁止する意味ではない。更新を意図的で追跡可能にする。
- **安全性**は、作業を遅くする意味ではない。安全な経路を最短にする。
